# ドキュメント作成ガイド

仕様書・設計ドキュメントの書き方と配置ルール。

---

## ライフサイクル

```
specs/（実装前の仕様書） → 実装 → docs/（永続アーカイブ）
```

- **specs/** … 実装を開始する前に書く仕様書（SPEC駆動の起点）
- **docs/** … 実装完了後に specs/ から昇格させた確定仕様（永続保管）

---

## 仕様書（docs/）

### 目的

実装済み機能の**設計判断・実装の根拠**を残す。「このコードがなぜこうなっているか」を将来の自分や Claude が参照できるようにする。

### docs/ サブフォルダの役割

| フォルダ | 格納するもの | 例 |
|---|---|---|
| `features/` | 機能単位の実装仕様（画面・コンポーネント・ビジネスロジック） | `auth-doc-01.md`, `dnd-reorder-doc-01.md` |
| `tech-spec/` | 実装中・実装後に発見した技術的 Tips・知見（ランタイム挙動・ライブラリの落とし穴など） | `cloudflare-workers-runtime-doc-01.md` |
| `rules/` | 実装前に参照すべきドメイン固有のルール（UI・UX・DB・認証など）。`rules/` と違い自動展開されないため実装時にオンデマンドで読む | `ui-rules.md`, `supabase-auth-rules.md` |
| `product-requirements/` | プロダクト要求定義（何を・誰のために作るか） | `product-requirements-doc-01.md` |
| `repository-structure/` | ディレクトリ構成・ファイル配置ルール | `repository-structure-doc-01.md` |

新しいカテゴリが必要な場合はサブフォルダを追加し、このテーブルに定義を追記する。

### 配置ルール

```
.claude/docs/
└── <カテゴリ>/
    └── <機能グループ>/       # features/ のみサブグループを持つ
        └── <機能名>-doc-<連番>.md
```

- カテゴリは上記テーブルのフォルダ名を使う（`features/` が最も頻繁に使われる）
- `features/` 配下は機能グループ単位でサブディレクトリを切る（例: `todos/`, `user-management/`）
- 機能名はケバブケース（例: `dnd-reorder`, `ai-todo-agent`）
- 連番は 1 始まりの整数（例: `01`, `02`）
- 同じ機能の仕様が改訂されたときは連番を上げて新ファイルを作る（旧ファイルは残す）

**例:**

```
.claude/docs/
├── features/
│   ├── todos/
│   │   ├── dnd-reorder-doc-01.md
│   │   ├── ai-todo-agent-doc-01.md
│   │   └── todo-crud-doc-01.md
│   └── user-management/
│       ├── user-management-doc-01.md   # 初版
│       └── user-management-doc-02.md   # 改訂版
└── product-requirements/
    └── product-requirements-doc-01.md
```

### 仕様書の構成テンプレート

```markdown
# <機能名> 実装仕様書

最終更新: YYYY-MM-DD  
ステータス: 完了 / 進行中

## 概要

<機能の目的を1〜2文で>

## 設計方針

| 項目 | 決定内容 | 理由 |
|---|---|---|
| ... | ... | ... |

## 実装ステップ

### Step 1 — <ステップ名>
対象: `src/...`
- 変更内容

### Step 2 — ...

## 関連ファイル

\`\`\`
src/
└── ...
\`\`\`
```

## フォルダ索引（INDEX.md）

`docs/` 配下の末端フォルダ（実ファイルが直接置かれるフォルダ。例: `features/<機能グループ>/`・`rules/`・`tech-spec/`・`migrations/`）、`.claude/skills/`、`.claude/rules/`、および `.claude/agents/` には、フォルダ内のファイル一覧とその役割をまとめた `INDEX.md` を置く。`.claude/skills/`・`.claude/rules/`・`.claude/agents/` はサブフォルダを持たないため、フォルダ自体を1つの末端フォルダとして扱う。

### 目的

サブフォルダの中身は自動的にはコンテキストに読み込まれないため、既存機能に関わる作業を始める前に `find`/`ls` で一覧を取り、さらに中身を推測するために個々のファイルを開く、という手戻りが発生していた。`INDEX.md` を1回読むだけでフォルダの中身と役割を把握できるようにする。

### 形式

| ドキュメント | 役割 |
|---|---|
| <連番を除いたベース名> | <1行で役割> |

同じベース名で連番違いの複数ファイル（例: `user-management-doc-01.md` / `user-management-doc-02.md`）は1行にまとめる。最新版はCLAUDE.md既定（最も連番が高いファイルを参照）で判別できるため、索引側に最新版番号は明記しない。

連番サフィックスを持たないファイル（例: `docs/rules/` 配下の各ルールファイル）は、ファイル名をそのままベース名とする。

### 維持管理

`INDEX.md` の作成・更新は `doc-push-agent` が行う（[.claude/agents/doc-push-agent.md](../agents/doc-push-agent.md) 参照）。

## 修正・変更の記録（docs/ への追記）

新規機能の仕様書とは別に、既存機能へのバグ修正・仕様変更も docs/ に記録する。

### 修正の分類と記録方針

| 種別 | 定義 | 記録方法 |
|---|---|---|
| **バグ修正** | 既存仕様の実装が仕様から外れていた | 本文はそのまま、既存 doc の末尾「修正履歴」に追記 |
| **軽微な仕様変更** | 振る舞いを変えたが、doc の構成・設計方針には影響しない規模 | **本文を現行仕様に書き換え**、あわせて「修正履歴」に経緯を追記 |
| **大幅な仕様変更** | 設計方針・構成が変わる規模 | 連番を上げた新ファイルを作成し、変更前後を明記 |

**軽微な仕様変更での本文書き換えについて**

軽微な仕様変更で本文を書き換えるのは「doc を読めば現行仕様が分かる」状態を保つため。変更の経緯・旧仕様は修正履歴側に残るので情報は失われない。

「本文は旧記述のまま修正履歴の注記だけで現行仕様を示す」方式は禁止。読み手が本文と履歴を突き合わせる必要が生じるため。

> 「いつ修正したか」は `git log` が担う。docs に残すのは「なぜ変えたか・何を変えたか」の設計判断のみ。

### 例外: 総括 doc は連番を上げず in-place 更新する

ある領域の**現況**をまとめた総括 doc（例: `docs/features/auth/auth-architecture-doc-01.md`）は、上記の分類の対象外とする。連番を上げず、同じファイルを直接書き換える。

対象の見分け方: 独自の設計判断を持たず、他 doc へのポインタと現在状態の事実だけで構成されているもの。設計判断はすべて個別 doc 側にあるため、in-place 更新で失われる情報はない（履歴は `git log` が持つ）。連番を上げると「現行の総括 doc がどれか」が分からなくなり、索引としての役割を果たせなくなる。

**更新のタイミング（重要）:** その領域に触れる変更と**同じ PR / doc-push の中で**総括 doc を更新すること。別タスクへ送らない。

> 理由: 個別 doc は連番を上げて新しくなるが、総括 doc は連番が上がらないため、意識的に直さない限り古いまま残り続ける。しかも総括 doc は「ここを読めば現況が分かる」と自称しているため、陳腐化すると個別 doc の陳腐化より害が大きい。

総括 doc には「まだ判断していない事項」を明記してよい。ただし判断が付いた時点で、その判断を下した変更と同じ PR / doc-push の中で書き換えること。

### 修正履歴の追記テンプレート

バグ修正・軽微な仕様変更の両方で使用。既存 doc ファイルの末尾に追加する：

```markdown
## 修正履歴

### <修正タイトル>（YYYY-MM-DD）   ← when

**種別:** バグ修正 / 仕様変更（軽微）  
**対象ファイル:** `src/...`

**変更内容:** <何をどう変えたか（what / how）>  
**理由:** <なぜ変えたか（why）>
```

**記述例:**

```markdown
### 一覧の既定の並び順を作成日時の昇順から降順に変更（2025-12-25）

**種別:** 仕様変更（軽微）  
**対象ファイル:** `<一覧を取得するモジュール>`

**変更内容:** 一覧取得時の既定の並び順を作成日時の昇順から降順に変更。  
**理由:** 新しく追加した項目が一覧の末尾に入り、ユーザーが見つけられないため。
```

**既存 doc の過去エントリについて**

既存ドキュメントの過去エントリ（問題/原因/対応 形式）は書き直さなくてよい。新規エントリから上記テンプレートを使用すればよい。

### 大幅な仕様変更の新ファイルテンプレート

`<機能名>-doc-<連番+1>.md` を作成し、冒頭に変更サマリーを記載する：

```markdown
# <機能名> 実装仕様書

最終更新: YYYY-MM-DD  
ステータス: 完了  
改訂理由: <前バージョンから何をなぜ変えたか1文で>

（以降は通常の仕様書テンプレートと同じ）
```

**自己完結ルール（必須）**

改訂版ドキュメントは前バージョンのすべての内容を含む自己完結ファイルとして作成する。「src/ の構造は doc-01 を参照」のような前バージョンへのコンテンツ参照は書かない。

> 理由: バージョンが増えると内容が複数ファイルに分散し、まとめ直すコストが増大するため。改訂時は前バージョンを読んでコピーしてから新規内容を加筆する。

## 情報の出所を明記する

技術ドキュメントに複数の情報源が混在する場合、**公式確認済みの情報**と**実装経験由来の知見**を明示的に分けて記載する。

```markdown
**公式ドキュメント確認済み：**
- ...

**実装経験由来（公式未確認）：**
- ...
```

### 理由

- 公式ドキュメントで確認できた事実と、実際に動かして得た知見は信頼性が異なる
- 将来読み返したとき「これは仕様として保証されているか」「仕様変更で変わり得るか」を判断できるようにする
- 公式未確認の知見を否定するのではなく、出所を明示することで読み手が適切に評価できるようにする

## rules/ との使い分け

| フォルダ | 用途 |
|---|---|
| `rules/` | **常時適用される制約・規約**（セッション開始時に自動展開される。どの作業でも常に有効なルールのみ置く） |
| `docs/rules/` | **ドメイン固有のルール**（実装時にオンデマンドで参照。UI・UX・DB・認証・テストなど作業ドメインが限定されるもの） |
| `docs/tech-spec/` | **実装中・実装後に発見した技術的 Tips・知見**（経験則・ランタイム挙動・ライブラリの落とし穴） |
| `specs/` | **実装前の仕様書**（SPEC駆動の起点） |
| `docs/features/` | **確定済み実装仕様のアーカイブ**（設計判断・完了した実装の根拠） |
| `skills/` | **やり方の手順**（再利用可能なボイラープレート・フロー） |

## ドキュメントの公開範囲と配置先

ドキュメントを作成するとき、**センシティブ情報を含むか否か**で配置先を分ける。

| 配置先 | git 管理 | 格納するもの |
|---|---|---|
| `.claude/docs/` | **追跡対象**（コミットされる） | 設計判断・実装根拠・アーキテクチャ方針。ドメイン固有の秘匿情報を含まないもの |
| `docs/`（ルート） | **gitignore**（ローカル限定） | DNSレコード・APIキー設定手順・メールアドレス・Supabase設定値など、ドメイン固有の設定情報を含むもの |

### センシティブ情報の判断基準

以下を1つでも含む場合は `docs/`（ルート・gitignore）に置く：

- 独自ドメイン名・サブドメイン
- メールアドレス
- DNSレコードの値
- 外部サービスの設定値（SMTP設定・APIキー発行手順など）
- その他、リポジトリ外に漏れると困る情報

上記を含まない純粋な設計ドキュメントは `.claude/docs/` に置いてコミットする。
