# agent 定義ファイルの作成ガイド

`.claude/agents/*.md` を新規作成・変更するときの規約。

## リポジトリパスのハードコード禁止

agent 定義ファイル内で作業対象リポジトリのパスを記述する場合、絶対パスをハードコードしてはならない。

理由: このプロジェクトはテンプレートとして他プロジェクトに切り出されることがあり、ハードコードだとコピー先で動作しない・手動置換が必要になる。

悪い例:
```
リポジトリ: /workspaces/cloudflare-actions/ai-todo
```

良い例:
```
リポジトリ: 起動時のカレントディレクトリ（`git rev-parse --show-toplevel` で解決されるプロジェクトルート）。固定の絶対パスへの `cd` は行わないこと。
```

## kit 配布対象の agent には特定スタック・ツール・サービス名を書かない

`workflow-kit`・`stack-kit` 経由で他のプロジェクトへ配布される agent 定義（例: `kit-push-review-agent.md`）には、特定のフレームワーク・BaaS・デプロイ環境・インフラ提供者・リポジトリの名前を含めてはならない。

理由:
- 配布先プロジェクトが同じスタックとは限らず、定義は配布先で意味をなさなくなる
- core 側 CI の混入検査（grep に基づく）に毎回ヒットし、例外宣言トレーラが常態化する。検査メカニズムを形骸化させる

対応: 層やスタック固有の語彙（技術名・プロジェクト名など）は、呼び出し側（スキル）が**引数として渡す**。agent 定義は層に無知なまま判定ロジックだけを持つ。

悪い例（kit 配布対象）:
```markdown
| 引数 | 例 | 役割 |
|---|---|---|
| `stack_term` | `["<フレームワーク名>", "<BaaS 名>", "<デプロイ環境名>"]` | 対象スタック固有の用語一覧 |
```
→ 具体的な技術名が agent ファイルに埋め込まれており、配布先で無意味になる

良い例:
```markdown
| 引数 | 役割 |
|---|---|
| `stack_term` | 対象スタック固有の用語一覧。呼び出し側が提供 |
```
→ agent は層に無知。具体値は呼び出し側が管理
