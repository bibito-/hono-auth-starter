# マージゲート（main への統合前ルール）

main にマージできる変更の **Definition of Done**。TDD の内側ループ（Red→Green→Refactor）とは別レイヤーの「統合境界のルール」。`tdd-workflow.md` / `spec-workflow.md` の両方からここを参照する。

---

## 位置づけ

```
TDD（内側ループ）        : Red → Green → Refactor   … 1変更の作り方
マージゲート（外側ループ）: grill → スモーク → 公式照合  … main へ統合してよいかの判定
```

各フェーズ／各段階（PR）の main マージ前に、以下3点を満たすこと。

| 要素 | 何を担保するか | 適用 | 出所 |
|---|---|---|---|
| **grill ゲート** | 開発者（ユーザー）が設計判断とその帰結を理解しているか／知識共有 | **選択式（既定オン）** | [[feedback-merge-gated-grill]] |
| **スモーク** | デプロイ後の実環境で重要経路が実際に動くか（モックでは検出不能） | 該当変更なら常時 | [[feedback-verify-prod-secrets]] |
| **公式照合** | 設定・API を記憶でなく公式ドキュメントで裏取りしたか | 該当変更なら常時 | documentation-guide「情報の出所を明記」 |

## grill ゲートの適用は選択式（既定オン・一度だけ決める）

grill ゲートは変更の重さで要否が変わるため、タスク開始時に1度だけ選ばせる。スモーク・公式照合は grill とは別軸で、該当する変更なら常に適用する。

```
このタスクのマージ前 grill ゲートをどうしますか？
A) 適用（既定・推奨） … 各 PR の main マージ前に理解度 grill を実施
B) スキップ … 軽微変更（リネーム・ドキュメント・自明な修正）で理解確認が不要なとき
```

- **決めるタイミング**: spec があれば `/spec` の Step 2-0、無ければ `/tdd` 開始時
- **記録**: 選択を `specs/proceed.md` と `steering/current.md` の「運用ゲート」に書く（例: `grill-gate: 適用` / `grill-gate: スキップ`）
- **再質問しない**: 既に記録があれば後続の `/tdd` で再度聞かない。その記録に従う
- **既定は A**。判断が記録されていない場合は A（適用）として扱う

---

## grill ゲートの原則

### 目的は3つ

1. **理解度の確認** — 開発者が「我々が下した設計判断・トレードオフ・その帰結」を理解しているか
2. **知識共有** — grill を通じて開発者と知識を揃える。テストであると同時に共有の場でもある
3. **理解の補強（spec への還元）** — grill の回答結果を基に、開発者が今後押さえておくべき点を spec の「押さえるべき要点」節に追記する。**一度間違えた箇所は再び間違えやすい**ため、後で見直したときに理解を促す装置として残す（後述）

### 出題してよいもの

- 我々が下した**設計判断**とその**理由・トレードオフ**
- その判断の**帰結**（例: フォールバック設計が本番を no-op にする理由、env ゲートの独立性が招く挙動、不可逆な操作の復旧コスト）

### 出題してはいけないもの（重要）

- **外部プラットフォーム/ライブラリのドメイン知識**で、開発者が設計判断から導けないもの（例: 「Vercel が `VERCEL=1` をビルド時に自動注入する」というプラットフォーム仕様）
- 理由: ユーザーが元々知らないドメイン領域は「絶対に分からない時がある」。それをいきなり問題に出すのは理解度テストとして無意味で、共有の目的にも反する
- **代わりにこうする**: ドメイン知識は documentation-guide の「情報の出所を明記」に従い、`📌 ドメイン知識（◯◯公式仕様）` の**注意書きとして提示し、こちらから説明する**。説明したうえで「その帰結」を問うのは可
- 詳細: [[feedback-grill-scope-domain-knowledge]]

#### 核心でない設問を出さない（重要）

- **出題者が既に答えを分かっている事柄を「言わせるだけ」の設問**は出さない。理解度テストとして無意味で時間の無駄
- **既に合意・正解に達した結論の言い換え再確認**を出さない。納得に達したら次へ進む
- 出してよいのは、**回答者と出題者の双方にとって核心で、まだ合意が取れていない論点**だけ
- 詳細: [[feedback-grill-concise]]

#### 設問を「残したい結論」から逆算しない（最重要・本来の役割の喪失）

- **失敗モード**: 先に「spec に残したい結論」を持ち、それを言わせるための問いを逆算で組む。これは「解答のための問い」ではなく **「問いを出すための回答」** で、grill を theater 化させる（誘導的・束ね型の設問になる）。
- **なぜ起きるか**: 本ゲートは「grill 合格 → 要点を doc に残す」を紐づけるため、**追記ネタを作るために設問を捻り出す**インセンティブが働く。これに乗ると grill 本来の役割を忘れる。
- **本来の役割（忘れるな）**: grill 対象は **「本質 ∧ doc に残したい ∧ ユーザーが補強を必要とする」点**の3条件すべてを満たすもの。問いはその発見の手段であって、記録の口実ではない。
- **失敗の症状**: 答えさせたい回答から逆算すると、設問が誘導的になり**余計な説明が過多**になる（「分かった気」を生み時間を浪費）。説明過多はこの逆算が起きているサイン。
- **正しい順序**:
  1. まず「誤解すると実害が出る、非自明な点」が**実在するか**を探す（コードベース・spec 探索を含む）。
  2. 在るなら最小の問いで確かめる。**無ければ grill を起動しない**。
  3. 設計メモを残したいだけなら Q&A の体裁を被せず**直接書く**。記録のために設問を作らない。

##### 過剰補正の禁止: 「文書化済み」を飛ばす根拠にしない（本質を殺さない）

- 上の逆算禁止を「spec に書いてある＝合意済み＝問わない」と運用すると、**本質が一切問われなくなる**（良い spec ほど本質を文書化しているため、全部スキップ対象になる矛盾）。これは誤り。
- **文書化済み ≠ 開発者が理解済み**。設計方針表に書かれた判断でも、その**帰結**の理解は別。grill の目的（理解度確認・知識共有）からすれば、**本質はむしろ最優先で問う対象**。
- **飛ばしてよいのは「開発者がこのセッションで既に理解を示した」点だけ**。「書いてあるから」は飛ばす根拠にならない。
- 禁止対象は**話題ではなく姿勢**: ダメなのは「記録したい結論から逆算した誘導・束ね型の問い」。本質を**素のクリーンな単問**で「本当に理解しているか」問うのは grill そのもので、推奨される。
- 詳細: [[feedback-grill-not-reverse-engineered]]

### 進め方

- **一問ずつ**。複数まとめて聞かない
- **設問は1〜2文**。前置き・トレース例・背景説明を重ねない（説明過多は意図をぼかし、「分かった気」で時間を浪費する）。前提共有が要るなら `📌` 一行で済ます
- **1設問＝1つの明確な核心回答**。期待する回答を一行で書けない設問は壊れている（複数解釈・因果が濁る）→ そのまま出さず、問いを割る
- **答えを先に出さない**（理解度テストモード。後述の「すり合わせ grill」とは逆）
- 回答が**明らかに誤っていたら掘り下げて再質問してよい**（Q1 が誤りなら Q1a・Q1b と枝を下げ、正しい理解に達するまで継続）。これは正しい仕組み。**禁止なのは「正解済みを言い換えて再確認」する方**
- トレース例・具体的な数値展開は、**回答が誤ったときの訂正フェーズでのみ**出す
- コードベースで確認できることは**自分で探索**し、ユーザーには聞かない
- 合格したら、確認できた論点を一覧で残す（steering / spec に記録）

### 合格後: spec への「押さえるべき要点」追記（目的3の実行）

grill 合格後、誤解しやすかった**設計判断の帰結**を spec の「## 開発者が押さえるべき要点（理解必須・grill 由来）」節に追記する。実装完了時に docs へ昇格され、将来この機能を触る開発者の参照になる。

**順序を逆転させない（最重要）**: 追記は「実在の理解欠落を grill で確認できた**副産物**」であって、grill の**目的ではない**。追記したいから問いを作る、の向きになったら theater 化のサイン（[[feedback-grill-not-reverse-engineered]]）。残すべき設計メモが在るのに確かめるべき欠落が無いなら、grill を経ずこの節へ**直接書く**のが正しい。

**3つのガードレール（必守）**

1. **記録対象を絞る** — 「**設計判断とその帰結**」で、かつ「**誤解しやすい/非自明**」な点だけ。些末な取り違えや自明な点は残さない（spec のノイズ肥大を防ぐ）
2. **ドメイン知識を「理解すべき点」として書かない** — プラットフォーム/ライブラリ仕様は `📌 ドメイン知識（◯◯公式仕様）` の注意書き側に置く（[[feedback-grill-scope-domain-knowledge]]）。要点節に雑学を混ぜない
3. **中立表現で書く** — 「**あなたが間違えた**」ではなく「**混同しやすい点**」と概念を主語にする。コミットされる文書に個人の誤答記録を残さない

### grill 適用を尋ねるときのメッセージ

grill ゲートの適用可否を尋ねるときは、**事前に意図を伝える**:

> あなたの回答を基に、spec を「あなたの今後の理解を助ける形」に追記する予定です（誤解しやすい設計判断の帰結を要点として残します）。

これにより grill が「揚げ足取りのテスト」でなく「理解を補強する協働」だと共有できる。

## grill の2モード（混同しない）

| モード | いつ | 推奨回答 | 目的 |
|---|---|---|---|
| **すり合わせ grill**（`rules/grill-me.md`） | `/spec` の意図確認時など、**決める前** | **各質問に推奨回答を添える** | 設計の枝を1つずつ潰して合意を作る |
| **マージゲート grill**（本ファイル） | main マージの**直前** | **答えを先に出さない** | 既に決めた判断の理解度を確認・共有 |

> すり合わせ grill は「提案して合意」、マージゲート grill は「理解の検証」。推奨回答の有無が逆になる。

## スモーク

デプロイ後に本番（または プレビュー）の実環境で、重要経路が実際に動くかを最小限の手動操作で確認する。網羅テストではなく「致命的に壊れていない」を確かめる実地検証。

> **デプロイは GitHub Actions が main マージで自動実行する（Worker / Vercel）。手動デプロイはしない・提案しない。** したがってスモークだけは構造上 **main マージ後**（CI のデプロイ完了後）に実施する段階で、grill・公式照合（マージ前）とは順序が異なる。「スモーク＝事前に手動デプロイ」と読まないこと。デプロイ単位は PR マージ。

- ユニットテストは env をモックするため、本番シークレット欠落・プラットフォームの env 注入漏れを**検出できない**（[[feedback-verify-prod-secrets]]）
- 新たに env 値を参照する変更・外部配信構成の変更は、スモーク項目を事前に列挙してから実施する

---

## 公式照合

設定ファイル・ミドルウェア・プラットフォーム API を書く前に、**記憶に頼らず公式ドキュメントを retrieval して裏取り**する。context 肥大を避けるため、必要な段階のものだけ参照する。照合結果は昇格 doc に「公式確認済み／実装経験由来（公式未確認）」を分けて記載する（documentation-guide）。

## PR 作成

TDD（Refactor 完了）後、マージゲートの各要素を踏まえたうえで PR を作成する。

- **タイトル**: 70 文字以内・変更の核心を端的に
- **本文 Summary**: 変更内容を箇条書き 1〜3 点
- **本文 Test plan**: スモークテストが必要な場合、**ユーザーが実施すべき手順をチェックリスト形式で列挙する**（別コンテナ・本番環境など Claude が直接確認できない操作が該当）
- base ブランチは `main`

```bash
gh pr create --title "..." --body "..." --base main
```

## マージ後の後処理（導通テスト Green 報告後に実施）

ユーザーから導通テスト（スモーク）の完了（Green）報告を受けたら、以下を順に実行する。

1. **spec を docs/ へ昇格**
   - `.claude/specs/<機能名>/` のファイルを `.claude/docs/features/<グループ>/<機能名>-doc-01.md` へ移動
   - 移動後、`.claude/specs/<機能名>/` ディレクトリを削除
   - 命名・配置ルールは [documentation-guide.md](../rules/documentation-guide.md) を参照

2. **steering/current.md を更新**
   - 進行中タスクのゴール・フェーズ・次のステップを削除
   - 「完了済み」に `- <機能名>（<日付>）` を追記
   - 「進行中タスク: なし」状態に戻す

3. **ローカル main を最新化**
   ```bash
   git checkout main && git pull origin main
   ```

## 適用

- 各フェーズ／段階（PR）の main マージ前に本ゲートを通す。段階分割している場合は**段階ごとに**実施し、その段階の論点に絞る
- steering/current.md の「運用ゲート」「次のステップ」にゲート実施を明記する
